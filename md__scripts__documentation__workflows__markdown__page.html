Adding a new Script

The following should be considered when adding a new script:

  1. Are you developing as part of a pattern (i.e. Observer Signals, Interface)?
  2. Are you exposing data for another script to utilize? If so, consider using the existing Observer Pattern Signals. If the data should be widely available, use an Interface (i.e. EntityInterface.cs) with getter/setters/adders/removers. There's no sense in calculating Velocity more than once.
  3. Are you creating a general schematic (Parent Class) for other classes to use? Consider creating a Template (i.e. AnimationTemplate.cs, ConfigurationTemplate.cs etc). You can then create child-classes that are specific/contain hard-coded values (i.e. KoboldSpearConfiguration.cs holds the specific health/mana/etc. of the Kobold Spearman).
  4. Are you creating something the performs active actions such as AI logic, sensory logic, etc.? Consider using a Controller (i.e. SensorController.cs). The controller naming convention helps separate active functionality from things like configurations and animations.
  5. Each dedicated Entity or Object should have it's own child-scene where scripts are attached (i.e. Player.tscn). This helps isolate things when searches through the node tree are performed, etc.
  6. Scripts that are dedicated to a type of Godot Node should inherit from that Node. For example, AnimationTemplate.cs class inherits from the AnimationPlayer class. This gives access to all of the functionality of the AnimationPlayer.
  7. Use getters and setters. This allows future use via Signals, but also makes the code self-documenting.

Generating Documentation via Doxygen

To re-generate documentation for the project, go to Documentation/Doxywizard and hit run on the final tab. If you update the Doxywizard's configuration, make sure to save it so that it doesn't have to be changed again.
